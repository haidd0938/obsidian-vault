# 老板语音/文字通信协议

> 适用平台：Telegram
> 规则来源：2026-05-04 老板明确要求 | 2026-05-05 补充：必须先语音后文字

## 🚨 黄金规则（必须严格遵守）

**你发语音 → 我回语音（优先）**
**你发文字 → 我回文字**

**入口条件（每次必须检查）：** 
- [ ] 用户输入是语音吗？ → **必须先用 text_to_speech 工具生成语音回复**
- [ ] 用户输入是文字吗？ → 用文字回复，不额外生成语音

**错误模式（已发生过，禁止）：**
- ❌ 收到语音 → 先回一段文字说明 + 最后附 MEDIA 音频文件 ← 老板讨厌这样，要求重来
- ✅ 收到语音 → 直接生成 TTS 语音回复，语音内容完整覆盖答案

**例外（要同时满足才成立）：**
- 老板明确说「用文字回」
- 或者 TTS 失败且 CLI 方案也无法挽救（这时道歉说明+发文字）

## 语音回复流程

1. 收到老板的语音消息（Telegram语音消息）  
2. 内容已被ASR转写成文字显示在对话中  
3. **直接调用 text_to_speech() 工具**生成中文语音，发给老板  
4. 回复中 `MEDIA:/path/file.ogg` 放在第一行，老板直接点开听  
5. 不需要额外写一段文字解释——语音本身就是回复

## 语音回复流程

1. 收到老板的语音消息（Telegram语音消息）
2. 内容已被ASR转写成文字显示在对话中
3. 回复时用 `text_to_speech()` 工具生成中文语音，发给老板
4. 如果 `text_to_speech` 失败，用 `edge-tts` CLI 生成 .mp3，然后通过 MEDIA: 路径发送

## 失败处理

如果TTS工具一时不可用：
- 尝试 CLI 方案（edge-tts 命令直接生成）
- 如果 CLI 也失败，明确告知老板"语音工具暂时故障，修好后补语音"，而不是擅自改成文字

## 不适用场景

- **紧急信息/警告**：可以先发文字告知情况，再补语音
- **长列表/结构化数据**：可以用文字补充，但主要回复仍是语音
- **老板明确说"用文字"**：以老板的临时指令为准

## 注意事项

- 语音要自然，不要太机械
- 语音长度适中，太长可以分两段发
- 中文发音要准确（Edge TTS 的 Xiaoxiao 语音表现良好）
